SOLID PATTERN AND DESIGN PATTERN



SOLID PATTERN
In software engineering, SOLID is a mnemonic summary of five design principles intended to make software designs understandable, flexible, and sustainable. Principles are part of many principles promoted by American software engineer and lecturer Robert C. Martin, first introduced in his 2000 paper Design Principles and Design Patterns.

SOLID stands for:
S - Single-responsibility Principle.
O - Open-closed Principle.
L - Liskov Substitution Principle.
I - Interface Segregation Principle.
D - Dependency Inversion Principle

LISKOV SUBSTITUTION PRINCIPLE
The idea of changing Liskov is represented by the letter L in the well-known SOLID abbreviation. The first policy statement is as follows: Methods that use basic class indicators should be able to use material from acquired classes without realizing it. At first, the idea seems simple. At first glance, it seems that it is no longer relevant when considering OOP's view of polymorphism. This theory is actually a warning to developers that polymorphism is powerful and deceptive: in the real world, the use of polymorphism often leads to a dead conclusion, so it should be used with caution. LSP is often summarized with a counter-example of Duck Test: “If it looks like a duck, quacks like a duck, but needs batteries – you probably have the wrong abstraction”

INTERFACE SEGREGATION PRINCIPLE
In accordance with the Interface Segregation Principle, the client should never be forced to use an unused connection, and clients should not be made to rely on unused methods. Clients should never be forced to use the visual interface they do not use, nor should they be forced to rely on methods they do not use. The goal of this policy is to eliminate unwanted code from Classes in order to minimize unforeseen problems when the Class is unable to perform an action. We have divided the activities into smaller groups so that the class can do only the actions they need.

DEPENDENCY INVERSION PRINCIPLE
The goal of dependency conversion is a way to freely connect software units. Standard dependencies built from high-level, policy-setting modules to low-level, dependency modules are distorted when this approach is followed, providing high-level independent modules without specifying the use of low-level modules. In principle:
A. High-level modules should not depend on low-level modules. Both should depend on abstractions.
B. Abstractions should not depend on details (concrete implementation). Details should depend on abstractions.



DESIGN PATTERN
The software design pattern is a common, reusable solution to a problem that often occurs in software design within a particular environment. It is not a finished design that can easily be translated into a source or machine code. It is a definition or metaphor for problem-solving that can be used in a variety of situations.
A - Design patterns are advanced processes that an editor can use to overcome common challenges while designing an application or system.
B - Design patterns provide answers to common issues, allowing the code to be manageable, expandable, and freely connected.
C - Solutions that handle a particular type of problem are named after the developer. That's how it all started.
D - The more one understands them, the easier it will be to solve all our problems.
E - The book Gang of Four (1994) made it popular.

WHY DO WE NEED DESIGN PATTERNS?
As Software Developers, we frequently assess our code based on criteria such as how clean, expressive, has a small memory footprint, and is quick. However, the most critical consideration, which we sometimes overlook, is that you should be able to simply adjust anything later. What you chose now might not be applicable tomorrow. And your code should be adaptable enough that changes are not prohibitively expensive. As a result, Design Patterns are best practices for covering such properties. The essence of Design Patterns, in my opinion, consists of the following six rules:

They Are Proven Solutions
Because Design Patterns are often used by developers, you may be confident that they function. Not only that, but you can be guaranteed that they were altered several times and that optimizations were most likely performed.
They Are Easily Reusable
Design Patterns describe a reusable solution that may be modified to solve a variety of specific situations. Because they aren't tied to a specific situation. Consider the Iterator Design Pattern, which is reusable across STL despite container and algorithm change. Iterators act as a glue between the container and the algorithm.
They Are Expressive
Design Patterns may elegantly describe a huge solution. The Visitor pattern, for example, is used to perform a new operation on a range/group of classes. As a result, the standard library followed a similar structure with a single function.

They Ease Communication
Developers that are knowledgeable with Design Patterns can communicate more readily about potential solutions to a given challenge. If you're part of a team of developers, agree on Design Patterns with your colleagues since they can help you solve problems more effectively. You should also follow similar practices when it comes to software maintenance, as it makes maintenance operations faster and more efficient.
They Prevent the Need for Refactoring Code
When an application is created with Design Patterns in mind, you may not need to rewrite the code later since applying the relevant Design Pattern to a specific problem is already an optimum solution. If such solutions are later updated, they may be applied effortlessly by any excellent software developer without causing any complications.
They Lower the Size of the Codebase
Design Patterns use less code than alternative solutions since they are generally beautiful and optimal. This is not always the case, since many engineers create extra code to increase comprehension.
